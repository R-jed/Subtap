# Subtap 安装分发与本地资源管理设计

## 产品目标

Subtap 面向不具备开发经验的 Apple Silicon Mac 用户。用户只需要理解三个动作：

1. 安装并启动 Subtap。
2. 选择音视频文件和字幕质量。
3. 等待任务完成并打开字幕。

Python、MLX、虚拟环境、模型路径、工作目录和中间文件默认不暴露。高级用户仍可通过 CLI 和配置文件控制，但不能增加普通用户的首次使用负担。

## 参考原则

借鉴 Mole 的产品逻辑：

- Homebrew 是普通用户的主安装入口。
- 安装、升级、卸载都有稳定命令和明确反馈。
- 发布产物按目标架构构建，带版本和 SHA256 校验。
- release workflow 自动更新独立 Homebrew Tap。
- TUI 把复杂命令组织成少量清晰入口。
- 下载、扫描和执行过程持续反馈进度。
- 危险操作必须说明影响并二次确认。
- 失败不能伪装成成功，错误必须给出可执行的恢复路径。

不照搬 Mole 的 Go/Bubble Tea 实现。Subtap 保留 Python、MLX 和 Textual，避免重写已经工作的 pipeline 与 TUI。

## 支持边界

- 正式支持 Apple Silicon Mac。
- 不承诺 Intel Mac、Linux 或 Windows。
- Homebrew 是普通用户主入口。
- PyPI 保留为开发者安装与依赖分发入口，不作为普通用户文档的首选路径。
- 模型不进入 Homebrew 包或 Python wheel，由 Subtap 自己管理。
- macOS 最低版本沿用当前项目声明，发布前必须在真实干净机器验证；未经验证不得扩大兼容范围。

## 用户安装体验

### 安装

普通用户文档只展示：

```bash
brew install r-jed/tap/subtap
subtap
```

首次执行 `subtap` 自动进入初始化向导，不要求用户先阅读配置文档或执行多条模型命令。

### 首次初始化

初始化向导按顺序完成：

1. 确认设备是 Apple Silicon Mac。
2. 检查可用磁盘、内存、FFmpeg 和 MLX 运行能力。
3. 根据机器条件推荐“快速”或“高质量”模型。
4. 显示下载大小、目标目录和预计耗时。
5. 下载模型并校验文件完整性。
6. 运行一次最小离线自检。
7. 成功后直接进入 TUI 首页。

任何步骤失败都停在当前步骤，保留已完成下载，并显示“重试”“切换下载源”“查看详情”三个明确动作。

### 更新

```bash
brew upgrade subtap
```

程序升级与模型升级相互独立：

- `brew upgrade` 更新程序。
- TUI“模型管理”更新模型。
- 程序启动时只提示兼容性要求，不自动下载大文件。
- 模型格式需要迁移时，先验证磁盘空间，再下载新版本；成功前保留旧版本。

### 卸载

```bash
brew uninstall subtap
```

Homebrew 卸载只删除程序，不删除模型、热词、文稿索引和历史任务。用户资料必须通过 TUI“清理本地数据”或显式命令删除，并在执行前列出目录与预计释放空间。

## Homebrew 与发布设计

### 可复用的 Mole 发布模式

Subtap release workflow 应复用以下结构：

1. 在目标 Apple Silicon macOS 环境执行格式、类型、测试和真实安装检查。
2. 构建带明确版本的发布产物。
3. 生成 SHA256 和发布清单。
4. 上传 GitHub Release。
5. 自动更新 `R-jed/homebrew-tap` 中的 `Formula/subtap.rb`。
6. 在临时 Homebrew 环境运行 `brew install`、`brew test`、`brew audit`。
7. 所有验证通过后才允许发布。

### 分发实现约束

Python、MLX 和二进制 wheel 不能假设与普通纯 Python Formula 一样可移植。实现前先完成一个最小冷安装验证，比较：

- Formula 安装隔离 Python 环境；
- Cask 安装自包含运行时；
- Formula 安装轻量启动器，首次初始化安装锁定运行时。

选择标准依次为：全新 Apple Silicon Mac 可重复安装、无需用户理解 Python、升级可回滚、Homebrew 审计可通过、安装体积合理。验证结论必须记录后再固定 Formula 或 Cask；不能为了保留既定方案绕过 Homebrew 规则。

无论最终载体是什么，用户命令保持 `brew install r-jed/tap/subtap`，模型始终独立下载。

### 发布门禁

发布前必须全部通过：

- `black --check src tests`
- `mypy src/subtap`
- 全量 pytest
- wheel/sdist 内容检查
- `subtap doctor --release`
- Homebrew 冷安装与升级测试
- 1.7B 完整本地离线 pipeline
- TUI 首次启动人工验收
- 无网络模式生成字幕
- 卸载后用户资料仍完整

## 本地文件结构

沿用现有单一根目录 `~/.subtap`，不同时引入 XDG、Application Support 和多个历史目录。

```text
~/.subtap/
├── config.yaml
├── state.json
├── models/
│   ├── asr_0.6b/
│   ├── asr_1.7b/
│   └── aligner/
├── glossaries/
│   ├── default.yaml
│   ├── learned.yaml
│   └── imported/
├── manuscripts/
│   └── index.json
├── jobs/
│   └── <task-id>/
├── cache/
│   └── downloads/
└── logs/
```

目录职责：

- `config.yaml`：用户设置的唯一来源。
- `state.json`：首次启动、最近任务和 UI 状态，不承载用户配置。
- `models/`：模型的唯一存放位置。
- `glossaries/`：默认、学习和导入热词表。
- `manuscripts/index.json`：文稿名称、原文件路径和最近使用时间，不复制原文稿。
- `jobs/`：可恢复任务与必要中间产物。
- `cache/downloads/`：可安全删除的断点下载缓存。
- `logs/`：诊断日志，不包含原始音频或完整文稿正文。

最终字幕永远写入用户明确选择的输出目录，不放进隐藏目录。

## 模型管理

### 模型清单

程序维护一个版本化模型清单，每项包含：

- 稳定模型 ID；
- 展示名称与用途；
- 下载大小和最低磁盘要求；
- 支持的运行模式；
- 下载源及镜像；
- 必需文件与 SHA256；
- 与 Subtap 版本的兼容范围。

代码和 TUI 都读取同一份清单，禁止各自维护模型名称或必需文件列表。

### 下载行为

- 使用临时文件下载，校验通过后原子移动到正式目录。
- 支持断点续传、重试、Hugging Face 镜像和用户取消。
- 下载失败不破坏已安装版本。
- 启动任务前校验当前选择模型，而不是要求安装全部模型。
- 删除模型前显示释放空间；正在使用的模型不能删除。

### 面向用户的模型选择

普通用户只看到：

- 快速：下载较小、速度优先。
- 高质量：默认推荐，质量优先。

具体参数、量化名称和模型目录放在“查看详情”中。

## 热词管理

TUI 提供“热词库”：

- 新建热词表；
- 导入 YAML；
- 编辑、搜索、启用和停用；
- 查看自动学习的候选词；
- 将候选词确认后写入 `learned.yaml`；
- 导出用户热词表。

默认规则：

- 用户导入内容不被自动覆盖。
- 自动学习结果与手工热词分开保存。
- 每次任务记录使用了哪些热词表，保证结果可追溯。
- 热词冲突必须展示，不静默选择。

## 文稿管理

TUI 提供“文稿库”，但不接管用户原文件：

- 用户选择 DOCX、TXT 或支持的文稿文件。
- Subtap 记录名称、路径、修改时间和最近使用时间。
- 任务开始时验证原文件仍存在且未被替换。
- 文稿移动或删除时提示重新定位。
- 默认不复制文稿正文到 `~/.subtap`。
- 任务日志只记录路径和摘要信息，不记录完整正文。

用户也可在单次任务中临时选择文稿而不加入文稿库。

## TUI 信息架构

### 首页

```text
Subtap
├── 新建字幕
├── 最近任务
├── 模型管理
├── 热词库
├── 文稿库
├── 设置
└── 系统检查
```

首页顶部只显示影响当前使用的状态：模型是否就绪、磁盘是否充足、是否存在未完成任务。

### 新建字幕向导

```text
选择音视频
→ 选择快速或高质量
→ 可选热词表
→ 可选参考文稿
→ 选择字幕语言与输出位置
→ 确认并开始
```

默认值来自上一次成功任务。高级参数收进单独区域，不与主流程并列。

### 任务运行页

只展示用户能理解和行动的信息：

- 当前阶段和整体进度；
- 已用时间；
- 当前模型；
- 暂停、取消和查看详情；
- 失败原因与恢复动作。

底层日志默认折叠。核心阶段失败必须停止，不允许以低质量假结果继续并显示成功。

### 完成页

- 打开字幕；
- 打开输出目录；
- 重新生成；
- 使用当前设置处理另一个文件；
- 查看任务详情。

## 任务与清理策略

- 每个任务使用独立 `task-id`，中间文件不共享。
- 成功任务默认删除可重建的大型中间音频，保留摘要和最终输出路径。
- 失败任务保留恢复所需文件与完整错误信息。
- TUI 显示任务占用空间并提供清理。
- 清理只允许删除 `jobs/` 和 `cache/` 中经过验证的路径。
- 清理模型、热词或文稿索引必须走独立确认流程。

## 迁移策略

首次启动检测现有 `~/.subtap`：

1. 读取旧配置并验证。
2. 生成迁移预览，列出移动和保留内容。
3. 先复制或原子重命名到新结构。
4. 验证模型和配置可用。
5. 成功后写入迁移版本。
6. 失败时保留原目录并提供恢复说明。

迁移不得自动删除未知文件。重复运行必须得到同一结果，不重复导入热词或文稿。

## 错误与可观测性

- 面向用户显示问题、影响和下一步，不显示无解释的 traceback。
- “查看详情”提供错误类型、阶段、模型和日志路径。
- 核心 pipeline 不吞异常，不生成伪造成功结果。
- 下载和模型校验记录来源、版本、校验值和失败原因。
- `doctor --json` 提供机器可读结果，`doctor --release` 只检查当前配置需要的模型。

## 开源项目开发纪律

### 产品决策过滤器

参考 Mole 的范围控制，每个新功能进入开发前必须回答：

1. 它是否直接帮助用户安装、生成字幕、管理资源或恢复失败任务？
2. 是否已有现成入口或系统能力可以复用？
3. 是否会增加普通用户必须理解的新概念？
4. 是否会扩大数据丢失、隐私或供应链风险？
5. 是否能够用自动化测试和真实用户流程验收？

不能明确改善常见用户动作的功能默认不做。不得因为技术上可行就增加后台常驻、云同步、账号系统、通知中心、插件市场或多平台兼容层。

### 单一验证入口

项目提供一个与 CI 完全一致的本地验证入口，例如 `scripts/check.sh`，统一执行：

- 格式检查；
- 类型检查；
- 单元与集成测试；
- 发布包内容检查；
- 敏感信息扫描；
- 用户目录安全检查。

开发者不需要记忆多套命令。CI 不允许执行本地无法复现的隐藏步骤。

### 变更纪律

- 每个修复先建立能复现用户问题的失败测试。
- 修改集中在共享根因位置，不在多个调用点重复打补丁。
- 一个提交只表达一个可验证目的。
- 不顺手重构无关代码，不引入未使用的配置和抽象。
- 核心行为、用户目录或发布渠道改变时，同步更新设计文档和用户文档。
- 合并前完成 Standards 与 Spec 两轴审查。
- 发布相关操作必须明确会影响 GitHub Release、PyPI 和 Homebrew 中的哪些渠道，不根据上下文擅自推断。

### 文件删除安全边界

Subtap 虽然不是清理工具，但会删除任务缓存、下载临时文件和模型，因此采用与 Mole 相同等级的路径保护：

- 所有递归删除进入同一个安全删除函数。
- 删除前解析绝对路径与符号链接。
- 拒绝空路径、相对路径、`..` 路径段、控制字符和用户主目录本身。
- 只允许删除 `~/.subtap/jobs/`、`~/.subtap/cache/` 和用户明确选择的单个模型目录。
- 最终字幕、用户原始媒体、原始文稿和热词表不进入自动清理范围。
- 面向用户的清理先预览路径与空间，再确认执行。
- 删除失败必须保留完整错误和未删除路径，不能报告全部成功。

任何新增的直接 `rmtree`、`unlink` 或 shell 删除命令都必须经过安全审查；测试覆盖主路径、符号链接、路径穿越和空变量场景。

### 测试环境隔离

- 测试必须使用临时 HOME 和临时工作目录。
- 自动化测试不得读取、修改或删除真实 `~/.subtap`。
- 模型下载测试使用小型固定夹具，不连接真实下载源。
- Homebrew 冷安装测试使用一次性环境。
- 破坏性测试默认 dry-run，并断言目标路径位于测试临时目录。
- 真实 1.7B pipeline 属于发布验收，不混入每次提交的快速测试。

### CI 与供应链安全

- GitHub Actions 默认只有 `contents: read`，仅发布步骤按需授予写权限。
- 所有 job 设置超时和并发取消，避免挂起或重复发布。
- 第三方 Actions 固定到完整 commit SHA，并保留版本注释。
- Dependabot 定期更新 Actions 和 Python 依赖，依赖更新必须通过完整验证。
- CI 使用专用工具扫描密钥、私钥和常见凭据；测试夹具不得包含真实密钥。
- 发布依赖使用锁定版本和可审查的更新流程，不在 release job 中临时安装 `latest` 工具。
- 构建日志不得输出 API key、用户路径中的敏感正文或模型访问凭据。

### 发布完整性与可回滚

- tag、Python 包版本、GitHub Release 和 Homebrew Formula 版本必须一致。
- 发布产物生成 `SHA256SUMS`。
- GitHub Release 生成构建来源证明；Homebrew 安装前校验哈希。
- 发布 workflow 验证最低 macOS 版本和 arm64 架构，不能依赖含义可能变化的模糊 runner 标签。
- 构建产物保留期明确，Homebrew 只引用正式 Release，不引用 CI 临时链接。
- Tap 更新失败时不影响已经发布的旧 Formula。
- 新版本安装验证失败时停止更新 Tap，不覆盖上一个可用版本。
- 保留上一版程序与模型兼容信息，提供明确的降级说明，不自动回滚用户配置。

### 性能与交互预算

- TUI 首屏不加载模型，启动应保持即时响应。
- 模型扫描和磁盘统计不得阻塞键盘交互。
- 长任务必须显示阶段进度，取消后在可预期时间内停止。
- CI 为启动、目录扫描和常见资源操作保留宽松但明确的性能上限；只有稳定复现的性能回归才阻止合并。
- 不把状态页变成指标墙，只展示能支持当前用户动作的信息。

### 开源协作资料

首次公开发布前补齐并保持同步：

- README：两条命令完成安装启动，并说明支持边界。
- CONTRIBUTING：本地验证入口、测试隔离和提交要求。
- SECURITY：漏洞报告方式、用户数据边界和供应链策略。
- 发布说明模板：用户可感知变化、迁移要求和已知限制。
- Issue/PR 模板：复现素材、系统信息、日志脱敏和验收证据。

文档必须描述当前真实行为，不能把计划中的功能写成已经可用。

## 开发阶段

### 阶段 0：恢复发布门禁

- 修复 README、Black 和现有测试失败。
- 修正 Doctor 只检查当前模型。
- 推送已完成的对齐 Fail Fast 修复。
- 建立单一本地验证入口，并补齐开发与安全文档框架。

### 阶段 1：分发验证

- 完成 Formula、Cask、轻量启动器三个冷安装原型。
- 选定通过 Homebrew 审计且用户步骤最少的载体。
- 固定 Apple Silicon 构建、校验和 Tap 自动更新。
- 固定 Actions SHA、最小权限、构建来源证明和回滚门禁。

### 阶段 2：统一资源层

- 固定 `~/.subtap` 目录契约。
- 建立单一模型清单和下载管理。
- 完成旧数据迁移与任务目录隔离。

### 阶段 3：TUI 用户流程

- 新首页与首次启动。
- 新建字幕向导。
- 模型、热词和文稿资源页。
- 最近任务、恢复和结果页。

### 阶段 4：发布验收

- 全新 Apple Silicon Mac 冷安装。
- 模型下载中断恢复。
- 无网络完整 pipeline。
- 升级、回滚和卸载验证。
- 普通用户人工验收。

## 验收标准

- 新用户仅靠 README 的两条命令进入可用 TUI。
- 首次启动无需理解 Python、MLX 或模型目录。
- 模型下载可恢复且校验失败不会污染安装。
- 热词和文稿可在 TUI 中管理，来源与任务可追溯。
- 所有任务输出位置清晰，临时文件可安全清理。
- Homebrew 安装、升级、卸载和审计通过。
- 发布产物具有 SHA256 与可验证构建来源。
- 自动化测试不会访问真实 `~/.subtap`，所有删除路径经过统一保护。
- 1.7B 本地离线 pipeline 与字幕质量不回归。
- 程序卸载后用户模型和资料默认保留。

## 非目标

- 不重写为 Go 或 Bubble Tea。
