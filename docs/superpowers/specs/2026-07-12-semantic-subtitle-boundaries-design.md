# 语义优先字幕断句设计

## 目标

修复最终字幕把完整句子或逗号后的完整短句错误合并的问题，同时让用户已有的 `max_chars`、`min_chars` 配置贯穿初次分句与最终导出。

验收示例：

- `一直是一机难求的状态。｜它叫做理光 GR4。`
- `它实际市场售价都已经到万元了，｜很难原价买到。`
- `但 GR4 又加了钱以后还是被抢爆，｜真的有点看不懂。`

## 根因

1. 初次分句使用模块常量 `45/15`，最终导出使用 `OutputConfig.max_chars/min_chars`（默认 `25/10`），同一文本经过两套不一致的阈值。
2. 最终导出的短句合并把 `min_chars` 当成强约束，可能跨越句末标点边界。
3. 逗号后的短评语虽然语义完整且存在语音停顿，但因短于 `min_chars` 被重新合并。

## 设计

### 单一配置源

- 用户入口保持现有 `--max-chars`、`--min-chars`，不新增同义参数。
- `OutputConfig.max_chars/min_chars` 是唯一配置源，默认值保持 `25/10`。
- `run_segment` 将这两个值传给 `segment_clean_segments`，替代初次分句中的固定常量。
- `run_final_exports` 继续使用同一组值。
- 参数必须满足 `max_chars > 0`、`min_chars > 0`、`min_chars <= max_chars`；无效配置直接报错，不静默修正。

### 边界优先级

1. `。！？.!?` 是硬边界，任何短句合并都不能跨越。
2. 逗号、顿号、分号是强候选边界；当边界处有明显停顿，允许右侧短句低于 `min_chars`。
3. `min_chars` 只阻止无语义碎片，不覆盖硬边界或带停顿的完整分句。
4. `max_chars` 仍是显示长度上限；没有自然边界时才使用词边界或固定长度切分。
5. 文本字符守恒、时间戳单调和不拆分词语仍是硬约束。

### 实现边界

- 只修改现有 segment/export 规则及参数传递，不新增第三套分句器。
- 复用现有 `_smart_split_v2`、jieba 词边界、word timestamps 和 CLI/config 能力。
- 不引入新依赖，不为特定句子添加词表或硬编码补丁。
- 热词误替换与英文字幕时间重叠作为独立问题，不纳入本次修改。

## 测试

### 快速反馈

- 在公开导出边界构造三个已确认文本，验证输出边界与预期完全一致。
- 验证句号硬边界不会被短句合并跨越。
- 验证逗号加停顿允许 6～9 字短评语独立。
- 验证无停顿的普通短碎片仍按 `min_chars` 合并。
- 验证自定义 `max_chars/min_chars` 同时传入 segment 和 export。
- 验证非法阈值快速失败。

### 完整验收

- 运行全量测试。
- 使用 1.7B、`quality + local + local-only` 完整离线处理 `高质量中文语音.mp3`。
- 人工审阅三处目标断句。
- 比较修复前后字幕：字符不丢失、词语不拆断、时间戳单调。
- 完成 Standards 与 Spec 两轴代码审查后再提交实现。

## 非目标

- 不修改 ASR 内容准确率。
- 不修复热词表中的错误替换。
- 不修复英文短语时间重叠。
- 不增加新的字幕风格预设或四套独立阈值。
